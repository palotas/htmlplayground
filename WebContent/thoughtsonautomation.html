A few thoughts on E2E automation (in no particular order)

Manual testing is still very important! - Yes please keep this in mind

Agile methodologies require frequent and instant feedback

Don't compensate missing unit tests with a heap of E2E tests

Drag and Drop Tools seem like a good idea, but often times lead to unmaintainable tests later due to lack of modularization

E2E tests are the most expensive type of tests. Therefore try to push as many tests as possible to lower levels

Running tests in parallel will reduce test execution times significantly and allows instant feedback

Open Source tools are cost efficient and allow you to be independent from tool vendors

Not everyone can automate; Test automation is software development

Automate the repetitive stuff

Do not treat automation as a side activity

Don't automate everything just because you can

If you want to scale, don't use record and playback tools

Think about mobile too

You get the biggest bang for the buck by automating regression testing

Build testability into your product

Include your developers, they are a great source of information

Don't measure the success of your automation by how many bugs the runs find

You will find most of the bugs while you automate and not while you run your tests

Keep your automation in synch with your production code

You cannot learn how to automate in a 1 day course but you can get started much quicker if you take one

Properly implemented test automation can reduce your manual regression effort to almost zero

Layout and usability are best verified by a human, they are very difficult to automate
